## 第二章节：交互篇 - 玩转 CLI 交互 (Slides 7-14)

### Slide 7: CLI 命令全景图
- **标题**：**基础 CLI 命令速查表**
- **视觉建议**：Icons + 命令描述。
- **核心要点**：
    - `claude`：启动会话。
    - `claude mcp`：管理上下文协议。
    - `claude agents`：查看子代理。
    - `claude update`：保持最新版本。
- **演讲稿**：在 Shell 层面，你可以直接调起多项管理功能。比如 `claude mcp` 就是开启第三方工具（如 GitHub、数据库查询）的钥匙。

### Slide 8: 风险控制：--dangerously-skip-permissions
- **标题**：**效率与安全的博弈**
- **视觉建议**：带有 [WARNING] 标识的警告框。
- **核心要点**：
    - 作用：绕过所有权限询问，全速运行。
    - **极重要：** 仅能在 VM 或 Docker 隔离环境中使用。
    - 宿主机风险：防止 AI 误删关键文件。
- **演讲稿**：这个参数很诱人，但也很危险。它让 AI 拥有“先斩后奏”的权力。除非你在 Docker 隔离环境里，否则严禁在主机会话中使用，安全第一。

### Slide 9: 核心斜杠指令 (I)
- **标题**：**Slash Commands：内场调度**
- **视觉建议**：命令/用途对照表。
- **核心要点**：
    - `/help`：帮助手册。
    - `/clear`：清除上下文污染。
    - `/compact`：手动压缩，节省 Token。
    - `/model`：按需切模型。
- **演讲稿**：在对话中，按下 `/` 键。`/clear` 特别有用，当你发现 AI 开始胡言乱语或者被之前的逻辑误导时，一张白纸重新开始往往是最高效的。

### Slide 10: 核心斜杠指令 (II)：项目管理
- **标题**：**自动化项目流：/init 与 /commit**
- **视觉建议**：从需求到 Git Commit 的流程图。
- **核心要点**：
    - `/init`：生成项目灵魂文件 `CLAUDE.md`。
    - `/commit`：AI 自动撰写符合规范的提交信息并执行。
- **演讲稿**：`/init` 建议在每个新项目根目录都跑一下。而 `/commit` 不仅是帮你写 Commit Message，它还会检查你的变更是否符合项目规范。

### Slide 11: 2026 新特性：/insights
- **标题**：**数据驱动自我提升：/insights**
- **视觉建议**：一个精美的个人报表截图（Token 趋势、工具使用率）。
- **核心要点**：
    - 生成路径：`~/.claude/usage-data/report.html`。
    - 内容：分析过去 30 天的 Token 消耗、常见误区建议。
- **演讲稿**：这是 v2.1.30 版本引入的新功能。它能生成一份 HTML 报告，告诉你哪一类任务你最耗钱，或者你在哪类问题上经常与 AI 进行无效拉锯，非常有价值。

### Slide 12: 时光倒流：Esc + Esc (Rewind)
- **标题**：**“后悔药”：系统级回滚**
- **视觉建议**：一个倒退的时钟图标。
- **核心要点**：
    - 操作：连按两次 Esc。
    - 效果：中断 Agent，弹出菜单回滚代码和对话历史。
- **演讲稿**：如果你看到 AI 正在疯狂修改你心爱的代码视图，甚至改错了，别慌，冷酷地连按两次 Esc。它是你的最高优先级紧急制动。

### Slide 13: 临时咨询：/btw
- **标题**：**插队咨询**
- **视觉建议**：两个并行的聊天气泡，一个大一个小。
- **核心要点**：
    - 场景：在处理复杂任务时，突然想问个不相干的小问题。
    - 优势：**不计入主会话历史**，不消耗主周期的 Token 复读成本。
- **演讲稿**：比如你正在重构，突然想查个 API 用法，用 `/btw`。这样这个咨询记录就不会污染你正在进行的重构上下文，省钱又省心。

### Slide 14: 交互小技巧
- **标题**：**高手进阶：快捷操作**
- **视觉建议**：代码片段演示。
- **核心要点**：
    - `! <command>`：直接在会话里跑 Shell 并捕获输出。
    - `@file`：显式指定文件，防止 AI 漫无目的地乱翻。
    - `Shift + Tab`：进入 Plan Mode。
- **演讲稿**：学会用 `!ls` 或 `!cat` 配合 `@file`，能让你的指令像手术刀一样精准。

## 第三章节：自动化篇 - /loop 的深度解析 (Slides 15-20)

### Slide 15: /loop 指令初识
- **标题**：**什么是 /loop：让 AI 成为你的后台守护进程**
- **视觉建议**：一个无限循环的 Logo。
- **核心要点**：
    - 语法：`/loop [任务] [频率]`。
    - 频率：支持自然语言描述（每 5 分钟、每小时）。
- **演讲稿**：有些事情不需要我们盯着。`/loop` 允许你开启一个定时任务，把重复性的监控或检查工作交给 Claude。

### Slide 16: 场景一：测试监控
- **标题**：**场景 1：重构期的质量守护者**
- **视觉建议**：代码修改与测试运行的同步动画。
- **核心要点**：
    - 示例：`/loop 运行 ! npm test 并报告失败 每 10 分钟`。
    - 价值：大型重构时，实时掌握“破坏”进度。
- **演讲稿**：你在写代码，它在后台每 10 分钟跑一次测试。一旦你的修改弄崩了现有的功能，它会立刻反馈给你，而不需要你手动切换去跑命令。

### Slide 17: 场景二：周期性代码审查
- **标题**：**场景 2：基于架构规范的“巡检”**
- **视觉建议**：警察巡逻的趣味图标。
- **核心要点**：
    - 示例：`/loop 分析最近 30 分钟更改是否符合 CLAUDE.md 规范 每小时`。
    - 价值：防止代码库随着开发时间推移慢慢“腐烂”。
- **演讲稿**：这就像是一个每小时查房的资深架构师，确保每一个新加的模块都没走样。

### Slide 18: 场景三：CI/CD 修复自动化
- **标题**：**场景 3：自主修复 CI 故障**
- **视觉建议**：GitHub Action 失败变成功的演变过程。
- **核心要点**：
    - 示例：`/loop 检查 PR #123 状态，若失败则尝试修复代码 每 20 分钟`。
- **演讲稿**：你可以把它挂在那里，让它盯着 CI。只要测试挂了，它就根据错误日志尝试本地修复并再次 push。

### Slide 19: 场景四：跨平台感知
- **标题**：**场景 4：信息汇总与同步**
- **视觉建议**：Slack、Email、任务清单的连接图。
- **核心要点**：
    - 示例：`/loop 汇总 Slack #dev 频道讨论并更新任务清单 每 2 小时`。
- **演讲稿**：配合 MCP 工具，Claude 可以变身成为信息管家，帮你从成千上万的信息中提炼出真正跟你这周任务相关的要点。

### Slide 20: 成本控制与局限性
- **标题**：**/loop 的生存守则**
- **视觉建议**：硬币（代表成本）与时钟（代表时间）。
- **核心要点**：
    - 限制：3 天自动过期（防 Token 浪费）。
    - 成本：建议设置较长频率。
    - 持久性：建议配合 `tmux` 使用，防止断连。
- **演讲稿**：别设太快，每分钟一次会让你的 Coding Plan 瞬间见底。建议从 30 分钟或每小时开始尝试。

## 第四章节：扩展篇 - 插件与 MCP 生态 (Slides 21-26)

### Slide 21: 插件 vs MCP 服务器
- **标题**：**外挂与桥梁：生态系统概览**
- **视觉建议**：拼图（Plugins）与 桥梁（MCP Server）的示意图。
- **核心要点**：
    - **Plugins**：内部功能扩展（如 UI 开发、文档同步）。
    - **MCP**：标准化协议，连接外部数据（GitHub、Slack、Database）。
- **演讲稿**：插件给 Claude 增加新技能，MCP 则给 Claude 打开通往外部世界（比如你的 GitHub 仓库）的大门。

### Slide 22: 三大市场分布
- **标题**：**去哪里找好工具？**
- **视觉建议**：三足鼎立的地图。
- **核心要点**：
    - `claude-plugins-official`：官方精品，如 `feature-dev`。
    - `superpowers-marketplace`：社区宝库，高级技能云集。
    - `voltagent-subagents`：代理专场，专门找“帮手”。
- **演讲稿**：通过 `claude plugin marketplace list` 可以查看你的所有能量来源。

### Slide 23: 必备工具 (I)：Context7 与 GitHub MCP
- **标题**：**实时知识与深度协作**
- **视觉建议**：大脑链接到云端文档。
- **核心要点**：
    - **Context7**：实时注入最新 API 文档（拒绝 2024 年的数据过期）。
    - **GitHub MCP**：搜索 Issue、创建 PR、读取深度代码。
- **演讲稿**：Context7 解决了 AI 知识过时的问题。如果你在用一个刚更新几天的框架，一定要装它。

### Slide 24: 必备工具 (II)：Feature-Dev
- **标题**：**官方出品：标准开发流插件**
- **视觉建议**：标准流水线图标。
- **核心要点**：
    - 功能：将一个 Feature 分解为 7 个标准阶段。
    - 优势：告别“直接上手改”，回归“先设计再编码”。
- **演讲稿**：这个插件强制 AI 按照成熟团队的规范工作：先分析、再设计、再编码。

### Slide 25: 必备工具 (III)：Playwright CLI
- **标题**：**UI 测试的省钱利器**
- **视觉建议**：Token 消耗对比柱状图（CLI vs MCP）。
- **核心要点**：
    - 推荐：使用 CLI 方案而非 MCP Server。
    - 数据：**节省约 4 倍 Token**。
    - 原理：将浏览器状态保存为本地 YAML/PNG 供 AI 读取，减少上下文堆积。
- **演讲稿**：在 2026 年，如果你要做网页自动化，一定要用 Playwright CLI 版本，这是社区证明最省钱的方式。

### Slide 26: 必备工具 (IV)：Claude-Mem
- **标题**：**解决“健忘”：持久化记忆**
- **视觉建议**：大脑内部的存储架。
- **核心要点**：
    - **90% Token 节省**：精准注入历史背景。
    - 可视化管理：`localhost:37777`。
    - 隐私自控：支持 `<private>` 标签。
- **演讲稿**：最痛苦的事是重开会话后，AI 忘了它是怎么修上个 Bug 的。Claude-Mem 让你的 AI 助手拥有“长久记忆”。

## 第五章节：实践篇 - Skills 最佳实践 (Slides 27-32)

### Slide 27: 什么是 AI Skills？
- **标题**：**定义你的工作流：Skills 机制**
- **视觉建议**：工具箱。
- **核心要点**：
    - 核心扩展机制。
    - 分类：通用技能 vs 项目专属技能。
- **演讲稿**：Skills 是封装好的工作流。熟练使用它们，可以让你从频繁敲打长 Prompt 中解脱出来。

### Slide 28: 通用技能 (I)：/simplify 与 /skill-creator
- **标题**：**精简与固化**
- **视觉建议**：剪刀剪掉冗余代码。
- **核心要点**：
    - `/simplify`：遵循 DRY 和 SOLID 原则的强制重构。
    - `/skill-creator`：当你发现自己在重复某个 Prompt 时，用它。
- **演讲稿**：写完代码，随手一个 `/simplify`。这不只是改格式，它会检查你的逻辑复用性和架构原则。

### Slide 29: 通用技能 (II)：/brainstorming 与 7 阶段流
- **标题**：**苏格拉底式提问：拒绝盲目编码**
- **视觉建议**：苏格拉底塑像。
- **核心要点**：
    - 流程：Intent -> Brainstorm -> Write Plan -> Execute.
    - 价值：通过提问澄清需求，防止走冤枉路。
- **演讲稿**：在动手前，先 `/brainstorm` 10 分钟。AI 会反向提问你，直到它完全理解你的业务场景。

### Slide 30: 灵魂文件：/claude-md-improver
- **标题**：**维护 AI 的“全局视野”**
- **视觉建议**：项目地图的更新。
- **核心要点**：
    - 使用时机：纠错后、新增模块后。
    - 价值：自动同步 `CLAUDE.md`，保证 AI 了解最新的项目地图。
- **演讲稿**：一旦你成功教育了 AI 某个特定知识点，记得跑一下这个技能，把它“固化”到项目的记忆库里。

### Slide 31: HotPlex 专属技能
- **标题**：**为我们的业务定制**
- **视觉建议**：HotPlex 品牌图标。
- **核心要点**：
    - `docker-container-ops`：一键冷热启动应用。
    - `hotplex-diagnostics`：全量健康诊断，带日志分析。
    - `hotplex-pr-master`：PR 全生命周期托管。
    - `hotplex-doc-sync`：文档防腐。
- **演讲稿**：这是我们自己开发的武器库。系统出问题了？直接丢给 `hotplex-diagnostics`，它看一眼日志就知道哪报错了。

### Slide 32: 如何管理这些技能？
- **标题**：**心中有数：技能列表管理**
- **视觉建议**：文件目录截图。
- **核心要点**：
    - 位置：`.agent/skills/` 下的 `SKILL.md`。

- **演讲稿**：技能也不是越全越好。定期清理不再适用的，并查看源码里的 `SKILL.md`，你会发现 AI 的很多“隐藏逻辑”。

## 第六章节：进阶篇 - 多代理 (Agent Teams) 编排 (Slides 33-40)

### Slide 33: 多代理协作的时代
- **标题**：**从“单打独斗”到“特种小组”**
- **视觉建议**：一群 AI 代理环绕一个核心任务的组网图。
- **核心要点**：
    - 核心：**Multi-Agent Orchestration**。
    - 目的：通过增加 Token 消耗来解决更复杂的问题。
- **演讲稿**：2026 年的一个标准操作是：你只有一个想法，但你有 5 个代理在同时为你开工。

### Slide 34: 模式对比：Subagents vs Teams
- **标题**：**选择你的武器**
- **视觉建议**：对比表格。
- **核心要点**：
    - **Subagents (子代理)**：上下文隔离，垂直通信，**省钱**，适合独立调研。
    - **Agent Teams (团队)**：共享任务列表，水平讨论，**高成本**，适合跨层攻关。
- **演讲稿**：如果只是想查个文档，派个 Subagent 即可。如果要重构整个鉴权层，那就得召唤团队。

### Slide 35: 开启实验性功能
- **标题**：**如何开启 Agent Teams？**

### Slide 37: /delegate 执行协议 (I)

### Slide 39: 对抗性调查模式

### Slide 40: 编排原则：Token 换时间

## 第七章节：架构篇 - 无头模式与 HotPlex 技术 (Slides 41-45)

### Slide 41: 什么是无头模式？
- **标题**：**脱离控制台：Headless Mode**
- **视觉建议**：彩色的 TTY 终端 vs 灰色的 JSON 数据流。
- **核心要点**：
    - 定义：无 GUI、无 TTY 交互。
    - 路径：`--output-format stream-json`。
    - 意义：不再跟“人”对话，而是跟“程序”对话。
- **演讲稿**：当我们想把 Claude 的能力集成到 Slack 机器人或者 CI 流程里时，就需要让它“无头”。它不再输出漂亮的动画，而是输出一行行结构化的 JSON。

### Slide 42: 非交互式快捷调用：-p 标志
- **标题**：**管道式 AI：-p 参数的高级用法**
- **视觉建议**：命令管道流向图。
- **核心和要点**：
    - 场景 1：`cat error.log | claude -p "修复它"`。
    - 场景 2：直接在脚本里调用并获取结果。
- **演讲稿**：`-p` 是最简单的无头用法。你可以把一段长长的日志直接 piping 给 claude，让它分析完直接吐出修复建议，一气呵成。

### Slide 43: HotPlex 的核心：热多路复用
- **标题**：**HotPlex 秘密：毫秒级响应的秘密**
- **视觉建议**：冷启动（走着走着要穿衣服） vs 热启动（已经在跑步机上）。
- **核心要点**：
    - 冷启动痛点：Node.js 启动 + 认证 (3-5s)。
    - HotPlex 方案：**常驻进程池 (Process Pool)**。
    - 成效：响应时间从秒级降至毫秒级。
- **演讲稿**：传统的 CLI 每次启动都要初始化。我们搞了个“进程池”，让几个 Claude 进程一直后台待命，你只要发请求，我们毫秒级分发。

### Slide 44: 关键机制：Session 持久化与分流
- **标题**：**如何让“热进程”不乱套？**
- **视觉建议**：Marker Files 与 ProviderSessionID 的映射图。
- **核心要点**：
    - **Marker Files**：即便重启服务，也能找回原有会话。
    - **环境隔离**：每个会话拥有独立的临时 CWD。
    - **PGID 清理**：杜绝 AI 启动产生的僵尸进程。
- **演讲稿**：我们通过 Marker 文件锁定了每个人的会话。即便我们的服务器崩了，重启后它依然认得你的上下文，非常可靠。

### Slide 45: 性能实测数据
- **标题**：**数据说话：性能提升报告**
- **视觉建议**：性能对比柱状图。
- **核心要点**：
    - TTFT（首字响应）：3000ms -> **150ms**。
    - 进程开销：下降 80%。
    - 极速体验：像调用本地函数一样调用 AI。
- **演讲稿**：这是我们引以为傲的成绩。20 倍的响应提升，意味着你可以随时随地、无延迟地与 AI 协作。
