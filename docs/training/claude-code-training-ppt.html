2. 什么是 Coding Plan？

Coding Plan 指代各模型提供商为开发者提供的 API 资源访问订阅方案。其架构关系如下：

  • Claude Code/OpenCode
    作为客户端交互控制层，负责接收终端指令并向底层下发代码变更。
  • Coding Plan
    提供底层语言模型的推理算力，决定了处理大规模代码逻辑所需的上下文窗口与 API 吞吐量 (Token)。

[ CHAP.03 ] 第三章：/loop 自动化工作流深度解析

/loop 是最强大的自动化指令之一，允许开发者以自然语言定义的频率循环执行特定的流水线任务。

1. 指令语法

/loop [任务描述] [执行频率]
示例：
  • /loop 检查当前分支的编译状态 每 5 分钟
  • /loop 总结过去一小时的 Git 提交并保存到桌面 每小时
注意：最小执行频率通常为 1 分钟。Claude 会将您的描述转换为内部定时调度 (Cron) 任务。

2. 核心使用场景 (1/2)

2.1 持续构建与单元测试监控

大规模重构时，持续运行测试套件并汇报。

场景：监控长时间运行的测试。

指令：
/loop 运行 ! pnpm test 并报告任何失败的测试用例 每 10 分钟

2.2 自动代码审查与质量巡检

团队协作时，定期验证架构与编码规范。

场景：周期性的自动化代码审查。

指令：
/loop 分析最近 30 分钟的代码更改是否符合 CLAUDE.md 中的架构规范 每小时

2. 核心使用场景 (2/2)

2.3 PR 与 Build 修复自动化

结合级联子智能体实现故障隔离与拦截式修复。

/loop 检查 GitHub PR #123 的集成测试状态，如果失败则尝试修复 每 20 分钟
2.4 / 2.5 集成汇总与部署跟踪

利用 MCP 跨平台汇总信息，跟踪环境发布进度。

/loop 汇总 Slack 最新讨论更新任务单 每 2 小时

/loop 检查生产环境 Pod 状态，直到实例 Running并通知 每 5 分钟

3. 高级技巧与限制

  • 持久性：/loop 任务通常在当前会话结束时终止。如需长期运行，建议配合 tmux 使用。
  • 有效期：循环任务通常在 3 天后自动过期，防止无意义的 Token 消耗。
  • 成本控制：频繁的循环会产生持续成本，必须在 claude config 中合理限制。
  • 查看状态：使用 /help 或 /statusline 观察循环运行情况。
提示： 初次尝试时，请先设置较长的频率！

[ CHAP.06 ] 第六章：Agent Teams 多智能体协作编排

在 v2.1.32+ 中，多代理协作 (Agent Teams) 成为处理复杂、跨模块工程架构任务的标准配置。

1. Subagents vs. Agent Teams

特性维度 Subagents (子调度) Agent Teams (团队协同)
协作模式 异步隔离处理
主从结构，单向指派任务。		 对等协同计算
平行结构，实时动态博弈。
上下文共享 完全隔离，互不干扰 共享任务列表，实时通讯锁
指令流向 垂直 (父 -> 子 -> 父) 水平交互 (Teammate <-> Lead)
核心场景 独立功能调研、BugFix 扫描 大型重构、BFF 架构全局设计
资源损耗 ● 极低 (精准召回) ● 较高 (通讯开销大)

2. 开启 Agent Teams (Experimental)

1. 版本要求：v2.1.32+。

2. 环境变量：

export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

3. 启动指令：

claude agents create --team --roles "lead,backend,qa"
tmux 模式：开启 teammateMode: "tmux" 观摩多屏协同。

[ CHAP.07 ] 第七章：无头模式与 HotPlex 面向生产的热多路复用

1. 什么是无头模式？

脱离标准 GUI 与 TTY，不依赖交互动效环境执行程序，采用纯结构化数据流通信。

核心价值： 程序化调用访问（IDE、机器人、CI）、全自动化流水体系、结构化 JSON 通信带来鲁棒性。

以 JSON 流模式启动：

claude --output-format stream-json --input-format stream-json

非交互式快捷调用 (-p 标志)

实战用例 1：代码解释

claude -p "分析 main.go 的并发逻辑"

实战用例 2：管道集成 (Pipeline)

cat error.log | claude -p "修复 bug"

实战用例 3：全自动化执飞行 (高危)

claude -p "运行 go test ./... 直到通过" --dangerously-skip-permissions

2. 高级玩法：HotPlex 热多路复用

传统无头 CLI 遭遇 2-5 秒“冷启动”开销。

原理解析：维持热预载控制池。HotPlex 托管预热进程并追踪活跃状态，将毫秒内将新请求分发入 Ready 的空闲热进程。

关键技术点

A. 会话持久化 (Marker)

即使重启通过 SHA1 Namespace 精确找到原始工程的 ~/.claude/projects 下会话并自动 --resume。

B. 全双工事件拦截

将复杂的 JSON 映射标准输出：拦截 thinking，对 tool_use 应用动态 WAF 安全校验。

C. 环境隔离与进程树守护

CWD 虚拟隔离目录环境。会话销毁时基于 PGID 进行全进程树剿灭，消除由于子节点启动的 DevServer 遗留问题。

性能对比总结

指标普通 CLI (冷启动)HotPlex (热多路复用)
首字响应 (TTFT)3s - 8s50ms - 200ms (~20x 提升)
进程控制每次重建抛弃高性能 IO 进程复用
会话恢复上下文手动处理或丢失防宕机自动持久化恢复
技术总结：热多路复用不仅保留了完整的工程编排能力，更达到了真正的 API 实时毫秒级响应能力，突破底座制约瓶颈。

[ SYS.PART.04 ] 第四篇：代码指挥官的思维进化

“不要教AI，要唤醒AI。上下文窗口是你的战场，你要控制迷雾，而不是制造噪音。”

1. 提示词四大反直觉技巧 (1/2)

  • 少即是多 (Less is More)
    SOTA大模型由于过强基础能力，过度微操反会框死。直接给出战略意图与边界约束。
    例：“写函数获取极值，务必鲁棒处理边界”，而非指点循环和比较大小。
  • 激活 vs 灌输
    不要用教的模式给准则，而是利用角色唤醒权威状态。
    例：直接设定“作为资深审计安全专家排查”，代替灌输一大堆测试条目。

1. 提示词四大反直觉技巧 (2/2)

  • 描述现象优先
    强制模型先输出“观察结果”，再给修复建议。极大降低幻觉。
  • 重复技巧
    对关键约束进行首尾复读。例如：“严格抽取报错行，严格抽取报错行”。

2. 打破“记忆”幻觉：上下文工程

模型没有全局持久记忆，而是依靠极易丢弃或失真的单次长文流转交互。

  • 拆解闭环：不要在一个超级回话里做全端包揽工作。用不同回话组合 /brainstorm 与 /feature-dev。
  • 管控心智信噪比：切换逻辑果断使用 /clear 清理战场。
